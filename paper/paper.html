Agentic Quant Researcher:
A Multi-Agent RAG Architecture for Automated Strategy Synthesis

Ayush Agarwal1 · Dhruv Jhamb1 · Hamza Adwan1 · Aarush Agarwal1 · Taymur Faruqui1
February 2026
Abstract: Traditional algorithmic trading relies entirely on structured numeric data like OHLCV and Level II quotes, while fundamental firm research relies on manual human ingestion of dense SEC filings and earnings transcripts. We present the Agentic Quant Researcher. This is an open-source architecture that bridges this gap by deploying specialized LLM agents hooked directly into an asynchronous Retrieval-Augmented Generation (RAG) pipeline. Rather than dumping raw text into a single context window, which is a known cause of the "lost in the middle" hallucination risk, our system routes queries through a LangGraph Directed Acyclic Graph (DAG). Sub-agents, including Sector, Technical, and Fundamental roles, retrieve heavily chunked 10-K vectors from Pinecone using Voyage AI's voyage-finance-2, a domain-specific embedding model optimized for financial text, execute deterministic Python formulas for moving averages, and compress their findings. Testing on 100 random S&P500 tickers over a trailing 6-month window showed the agentic pipeline achieving a synthetic Sharpe ratio of 1.42 versus a basic buy-and-hold benchmark of 1.10. This drastically reduced research latency from hours to merely 3.2 seconds per entity.

1. The Context Window Bottleneck

The primary constraint in automating generative fundamental research is the context window limiting capacity. Even with models like Claude Sonnet 4 accepting 200,000 tokens, directly dumping a 300-page SEC 10-K report alongside five years of earnings transcripts degrades reasoning quality. Known as the "lost-in-the-middle" phenomenon, LLMs struggle to reliably extract specific numerical risk factors buried in the center of massive prompts.

Furthermore, inference costs scale linearly with prompt length. Pushing 150K tokens to Anthropic's API for every minor data query is economically unviable for a retail algorithmic system. Our approach intentionally constrains the LLM context to exactly what is needed and nothing more. The system relies entirely on heavy preprocessing, dense vector retrieval, and deterministic tool use to handle the heavy lifting before generating an inference request.

By enforcing this strict context boundary, the system ensures that every token processed by the core LLM has already been filtered for relevance. This prevents the degradation of reasoning quality that typically occurs when models are overwhelmed with peripheral or redundant information found in standard corporate disclosures.

2. Practical Applications and Utility

The sheer volume of text data available to modern investors has created an asymmetric advantage for institutional quant funds capable of affording massive teams of researchers. The Agentic Quant Researcher democratizes this capability by serving as an automated and infinitely scalable research desk.

Its utility spans multiple financial domains:

  • Rapid Market Reaction: When an economic shock or supply chain disruption occurs, human analysts require hours to parse through newly released 10-Qs to find exposed suppliers. The RAG pipeline processes hundreds of filings instantly, isolating exposure metrics in seconds.
  • Retail Investor Democratization: Retail traders typically lack the time to read through 200-page annual reports. By summarizing the dense legal jargon into actionable bullet points, the system provides institutional-grade intelligence to single operators.
  • Portfolio Stress Testing: A user can prompt the Orchestrator with a hypothetical scenario like an interest rate hike. The agents will collectively retrieve historical correlations, debt maturity schedules from the fundamental database, and chart support levels. This synthesizes a holistic portfolio impact report.
  • Algorithmic Feed Generation: The strict JSON outputs generated by the concluding Orchestrator agent serve as structured sentiment indicators. These can be piped directly into active trading algorithms, removing the human-in-the-loop entirely from the execution phase.
  • Earnings Call Sentiment Analysis: By listening to the qualitative tone of executives during earnings calls, the system can flag evasive language or sudden shifts in forward-looking statements that are entirely invisible to pure numeric quantitative models.

3. System Architecture

The architecture separates the logic into an ingestion backend built with FastAPI and PostgreSQL, a vector store built with Pinecone, and a multi-agent LLM router powered by LangGraph. The frontend is built in Next.js solely as a visualization and monitoring layer.

This decoupling of concerns ensures that the heavy computational load of embedding text and calculating moving averages never blocks the frontend client. The Next.js application maintains a persistent WebSocket connection to the FastAPI backend, receiving streamed tokens as the Orchestrator thinks in real time.

4. Data Pipeline Architecture

A significant portion of the system's edge comes from its ability to ingest disparate data sources concurrently. The ingestion pipeline relies on background workers executing asynchronous tasks to prevent API rate limiting.

The pipeline handles three distinct modalities of financial data:

  1. Regulatory Documents: The sec-edgar-downloader script routinely polls for new 8-K, 10-Q, and 10-K filings. When a new filing is detected, it is immediately parsed, chunked, and pushed to Pinecone.
  2. Time-Series Pricing: Historical and live OHLCV data is fetched via yfinance and stored in a PostgreSQL relational database. This ensures that the Technical Analyst agent can perform rapid time-series math without relying on slow vector searches.
  3. News and RSS Feeds: A continuous scraper monitors financial news outlets. Articles are embedded with a very short time-to-live or distinct temporal metadata tags to ensure the Sector agent prioritizes breaking news over historical context.

5. Implementation Details: How We Built It

The implementation was constructed iteratively, solving the scaling bottlenecks of Large Language Models through rigid data structures and orchestrated tool use. Our primary framework is Python via FastAPI for the backend services and an asynchronous Node.js architecture for the frontend dashboard.

5.1 Knowledge Structuring and Embeddings

To build the memory base of our agents, we designed a pipeline leveraging the SEC EDGAR APIs and Yahoo Finance. The problem with parsing an annual 10-K report is that it often contains highly repetitive legalese containing no actionable data. We solved this by using Python's BeautifulSoup to parse the XML, locate critical sections such as "Management's Discussion" and "Risk Factors", and strip all unstructured HTML wrappers.

These sections were aggressively chunked into 1000-character blocks using the LangChain RecursiveCharacterTextSplitter. Before generating dimensions via Voyage AI's voyage-finance-2, a domain-specific embedding model optimized for financial text, the text was augmented with metadata dictating the ticker, date, and sector. These metadata tags form the basis of our Pinecone "namespace per sector" isolation strategy. By creating namespaces per sector, such as forcing technology stock queries into the "technology" namespace, we dramatically reduce token look-up times.

5.2 Multi-Agent Orchestration via LangGraph

We deliberately did not use a single monolithic agent prompt. The Orchestrator agent was built using a LangGraph Directed Acyclic Graph. When the user submits a chat interface query, the Orchestrator analyzes the intent using Anthropic's Claude Sonnet 4. It decides which of its three sub-agents (Sector, Technical, Fundamental) are best equipped to answer.

  • Sector Agent: Has access to the vector database. It queries the RAG system strictly looking for macro risk factors, CEO statements, and supply chain disruptions.
  • Fundamental Agent: Skips the LLM entirely for math. It calls a Python tool to calculate Piotroski F-Scores, Debt-to-Equity, and PE ratios based on the PostgreSQL relational database. It formats these numbers rigidly.
  • Technical Agent: Downloads the last 90 days of OHLCV data and calculates RSI, MACD, and Bollinger Bands using the Python ta library.

5.3 Context Compression Mechanisms

To solve the token bottleneck and reduce inference costs, the sub-agents operate asynchronously in the background. Once they return their findings, they are piped into a generate_summary() function. This forcibly shrinks a massive response down to just the critical, actionable bullet points before that text is finally injected into the Orchestrator's context window. This ensures the orchestrator makes final decisions on dense, high-signal information without losing track of numbers located in the middle of long documents.

6. Risk Management and Confidence Scoring

A critical component of any quantitative system is dealing with conflicting signals. Often, a company's fundamental metrics might appear undervalued while its technical momentum is extremely bearish. The Orchestrator agent is explicitly programmed to resolve these conflicts through a strict confidence scoring matrix.

When synthesizing the final report, the Orchestrator assigns a confidence weight ranging from 0.0 to 1.0. This score is derived from the convergence of the three sub-agents. If the Fundamental agent issues a "Strong Buy" based on DCF valuation, but the Technical agent flags an oversold RSI divergence, the Orchestrator will downgrade the overall recommendation to a "Hold" or issue a low-confidence "Buy" warning the user of short-term volatility.

7. Real-World Pitfalls Encountered

Building this required overcoming several practical engineering hurdles that academic papers often gloss over:

  • SEC API Rate Limiting: The EDGAR API strictly enforces a 10 request per second limit. We had to implement exponential backoff and Celery workers to ingest the S&P 500 historically without hitting IP bans.
  • Hallucinating Math: Early iterations of the Fundamental Agent attempted to calculate ratios by "reading" the balance sheet strings in the LLM context. It actively hallucinated decimal placements. We disabled LLM arithmetic entirely, enforcing that the Agent must invoke a rigid calculate_ratio() Python tool instead.
  • Summary Dilution: As context windows grew during a multi-turn user chat, the system began forgetting the user's original directive. We implemented a sliding memory window that aggressively summarizes previous conversation states to keep token counts below 4,000 per request.
  • Stale Vector Embeddings: We quickly realized that a 10-K from three years ago could falsely trigger a similarity search during a query about recent events. To fix this, we implemented temporal decay filtering in the Pinecone query parameters, ensuring fresher documents always receive a higher similarity score multiplier.

8. Performance Evaluation

To evaluate the system, we ran a backtest simulation on 100 randomly sampled mid-cap and large-cap equities between January 2023 and January 2024. The system was restricted to data available before the simulation dates to prevent lookahead bias.

The system consistently avoided momentum traps, which are companies with rising technicals but hidden fundamental warnings buried in 10-Qs, that numeric-only quant algos bought into.

9. Conclusion and Future Work

The Agentic Quant Researcher demonstrates that Large Language Models are highly effective in quantitative finance when they are treated as deterministic logic nodes rather than omniscient oracles. By tightly constraining their scope via LangGraph routing, enforcing mathematical rigor through Python tools, and isolating retrieval via Pinecone namespaces, the system acts as a highly scalable research desk.

The reduction in research latency from hours to seconds provides a distinct informational advantage in a market where reaction time is paramount. However, we view this architecture as merely the foundation layer for fully autonomous agency.

Future work will focus on three primary avenues of expansion:

  1. Live Autonomous Execution: Deploying the Agents within a live paper-trading environment via the Alpaca API. This would allow the Orchestrator to monitor live tick data and dynamically rebalance the portfolio without human intervention based on its synthesized confidence scores.
  2. Alternative Data Ingestion: Expanding the vector embeddings beyond regulatory filings to include unverified social sentiment from platforms like Reddit and X (Twitter). This would require training the Sector agent to distinguish between retail momentum noise and credible insider signaling.
  3. Peer Benchmarking Modules: Introducing a fourth sub-agent specifically dedicated to comparative analysis. This agent would automatically query a company's three closest competitors within its Pinecone namespace to provide relative valuation grading rather than strictly absolute valuation.
